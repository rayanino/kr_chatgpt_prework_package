# KR ChatGPT Pro Operating-System Package

## What this package is

This package contains the operating-system design for running KR inside ChatGPT Pro in a disciplined, high-rigor way.

It is not the real KR production project.

It is the package that will be used to create and harden the **pre-work project**, whose job is to validate, stress-test, tighten, and finalize the environment before the actual KR production project is launched.

This package exists so that ChatGPT Pro can later operate as:
- KR’s thinking owner
- creative product manager
- senior architect
- doctrine hardener
- systems director
- strategic reviewer

The goal is not merely to “set up a ChatGPT project.”
The goal is to create the strongest practical operating environment possible for long-horizon KR work.

---

## What this package is for

This package is meant to support two sequential projects.

### 1. The pre-work project
This project audits and improves the operating system itself.

Its job is to:
- review the package critically
- detect contradiction, redundancy, weakness, overengineering, or missing pieces
- validate launch readiness
- research ChatGPT/Projects/Pro usage where needed
- finalize what should and should not transfer into production

### 2. The production project
This is the real long-horizon KR strategic command project.

It should only be created after the pre-work project has approved launch.

Its job is to:
- harden one KR aspect per chat
- work one bounded subtask at a time inside each chat
- produce durable strategic outputs
- create repo-native artifacts
- prepare research requests and execution packets when appropriate

---

## What this package is NOT

This package is not:
- the final canonical truth about KR
- the final repo structure
- the final production upload set
- a replacement for the actual KR repo
- a substitute for validation inside ChatGPT Pro

It is a **structured draft operating system under audit**.

Some files will likely transfer into production with little or no change.
Some may be tightened, merged, reclassified, or removed during pre-work.

---

## Package philosophy

This package is built around a few core ideas:

- the environment should not limit performance
- ChatGPT Pro should be used as the main strategic mind, not as a passive assistant
- one chat should focus on one aspect only
- one bounded subtask should be advanced at a time
- the user should be able to operate mostly with short prompts like `Continue.`
- durable outputs matter more than good conversation
- the repo should accumulate structured project intelligence, not just chat history
- deep research should be used aggressively where justified, but not wastefully
- Codex and other implementation agents should receive bounded, reviewable work packets rather than vague strategy dumps

## How to use this package

### Step 1
Create a local folder for this package.

### Step 2
Place the 24 markdown files into that folder.

### Step 3
Add the deployment text files that will later be pasted into ChatGPT:
- global custom instructions
- pre-work project instructions
- pre-work kickoff prompt
- production project instructions
- first production chat opener

### Step 4
Use the `MANIFEST.md` file to understand:
- what each file is for
- which files are pre-work-only
- which files transfer into production
- which files are deployment-only

### Step 5
Buy ChatGPT Pro.

### Step 6
Create the **pre-work project**, not the real production project.

### Step 7
Upload the package there in the intended order and let the pre-work project audit and harden it.

### Step 8
Only after the launch gate passes, create the real KR production project.

## Package layers

This package is organized conceptually into three layers.

### Layer A — pre-work governance
These files govern the design, validation, and launch-gating of the operating system itself.

### Layer B — production operating system
These files define how the future KR production project should actually think and operate.

### Layer C — deployment materials
These files are practical launch aids and paste-ready texts for creating the projects and starting the first real chat.

This separation is important.
The production project should inherit the right operating system, not the entire meta-history of how the operating system was designed.

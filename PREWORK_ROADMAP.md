# KR Pre-Work Project Roadmap

## 1. Purpose of this roadmap

This roadmap defines the exact sequence by which the KR pre-work project must design and finalize the operating system for the future real KR project.

It exists to prevent setup drift, premature production work, redundant discussions, and vague progress.

This roadmap is not a backlog. It is a gated completion path.

The project is complete only when every required stage has passed its completion gate.

---

## 2. Governing rule

The pre-work project must proceed in a strict order.

A later stage may be explored early only if doing so is necessary to unblock an earlier stage, but no later stage may be considered finalized before its dependencies are finalized.

The sequence is designed to ensure that:
- the real KR project starts with a coherent operating system
- critical protocols are designed before they are needed
- instruction layers are built on top of a stable role model
- execution handoff systems are built only after the thinking system is defined
- the final production project can start cleanly and immediately

---

## 3. Completion philosophy

The goal is not to discuss every setup topic.

The goal is to finish every setup topic that is structurally necessary for the real KR project to operate at high quality.

Each stage must end with durable artifacts, not only reasoning.

Each stage must answer:
1. what problem this stage solves
2. what durable outputs it must produce
3. how those outputs will be used in the real KR project
4. what conditions must be satisfied before the stage can be marked complete

---

## 4. Roadmap overview

The pre-work project must complete the following stages, in order:

### Stage 0 — Baseline ingestion and operating assumption lock
Understand KR’s current state, current repo law, current bottlenecks, and what the pre-work project is allowed to challenge.

### Stage 1 — Role model and authority system
Define exactly how ChatGPT Pro is meant to behave in the future KR project.

### Stage 2 — Core instruction stack
Design the final instruction layers that will shape the model’s behavior.

### Stage 3 — Chat workflow protocol
Define how work happens inside the project and inside each chat.

### Stage 4 — Deep-research and review-control system
Define when deep research is used, how decisions are reviewed, and how quality is stress-tested.

### Stage 5 — KR aspect system
Define the complete aspect map that the real KR project will harden over time.

### Stage 6 — Repo artifact system
Define what files, registries, doctrine notes, templates, and other durable artifacts the real project should create and maintain.

### Stage 7 — Execution handoff system
Define how strategic outputs are turned into bounded implementation packets for Codex or other execution agents.

### Stage 8 — Production-project bootstrap package
Assemble the exact instructions, files, startup procedure, and launch sequence for the real KR project.

### Stage 9 — Pre-launch validation
Test the whole setup package against realistic use so that the real KR project does not begin with hidden setup flaws.

The project is complete only after Stage 9 passes.

---

## 5. Stage-by-stage plan

# Stage 0 — Baseline ingestion and operating assumption lock

## Purpose
This stage establishes a precise understanding of KR’s current state and draws a clean line between:
- what exists now
- what the repo currently believes
- what the pre-work project may revise, extend, or reject

Without this stage, the operating system could be designed on false assumptions.

## Required outputs
- a concise current-state summary of KR
- a list of existing durable repo laws that should remain binding
- a list of current assumptions that should be treated as challengeable rather than fixed
- a list of current bottlenecks relevant to the setup effort
- a statement of how the pre-work project should relate to the repo: anchored by it, not imprisoned by it

## Completion gate
This stage is complete only when the pre-work project can clearly answer:
- what KR currently is
- what control law already exists
- what is allowed to be rethought
- what parts of the current repo should constrain the future project and what parts should not

## Failure mode
If this stage is skipped or weak, the rest of the setup will either blindly obey the repo or ignore it irresponsibly.

---

# Stage 1 — Role model and authority system

## Purpose
This stage defines who ChatGPT Pro is supposed to be inside the real KR project.

This is the foundational behavioral layer. It determines whether the model acts like:
- a passive assistant
- a thoughtful collaborator
- or the acting thinking owner of KR

The intended answer is the last one, but it must be formalized carefully.

## Required outputs
- explicit role definition
- authority model
- initiative rules
- question-minimization rules
- escalation rules
- assumption rules
- uncertainty rules
- boundaries for decisions the model can make on its own
- boundaries for decisions that require user sign-off
- boundaries for decisions that require external evidence

## Completion gate
This stage is complete only when the future KR project could answer all of the following:
- what is the model’s role
- when should it decide rather than ask
- when must it escalate
- how should it treat uncertainty
- how much initiative should it take
- what kinds of ownership does it have

## Failure mode
If this stage is weak, the model will either become overcautious and annoying, or overconfident and unsafe.

---

# Stage 2 — Core instruction stack

## Purpose
This stage turns the intended role and behavior into actual instruction layers for ChatGPT Pro.

It must define the complete instruction system that will shape the production project.

## Required outputs
- final global custom instructions
- final production-project instructions
- any supplementary instruction files used as pseudo-skills
- instruction-layer separation logic:
  - what belongs globally
  - what belongs in the project
  - what belongs in attached files
- instruction conflict rules
- instruction brevity rules where needed
- behavior priorities when instructions compete

## Completion gate
This stage is complete only when the future production project’s instruction system is ready to paste in and use directly.

## Failure mode
If this stage is weak, the project will start with vague or conflicting behavioral guidance, leading to drift.

---

# Stage 3 — Chat workflow protocol

## Purpose
This stage defines how actual work happens inside the future KR project.

This includes both the macro structure and the micro loop.

The desired model is:
- one aspect per chat
- one subtask at a time inside the chat
- user can prompt with “continue”
- the model keeps advancing in bounded, deep increments

## Required outputs
- one-aspect-per-chat protocol
- one-subtask-at-a-time protocol
- “continue” workflow rules
- session progression model
- how a chat opens
- how a chat narrows its current subtask
- how a chat decides whether to continue the same thread or conclude
- how a chat ends with durable output
- rules for when a new chat should be opened instead of extending the current one

## Completion gate
This stage is complete only when the future KR project has a stable, repeatable, low-friction work loop that prevents chats from becoming broad, confused, or too shallow.

## Failure mode
If this stage is weak, the project will either become unfocused or overly fragmented.

---

# Stage 4 — Deep-research and review-control system

## Purpose
This stage defines how the future KR project uses high-cost cognition and external verification.

You explicitly want the project to make maximal high-value use of:
- deep reasoning
- deep research
- exhaustive review
- critique passes
- decision hardening

This stage formalizes that.

## Required outputs
- deep-research trigger policy
- criteria for when internal reasoning is sufficient
- criteria for when external research is required
- review protocol for major decisions
- adversarial critique protocol
- multi-pass hardening protocol
- decision confidence labeling system
- unresolved-question tracking rule
- evidence-strength categories
- escalation rule for high-stakes uncertain claims
- rule for when research should be deferred rather than immediately run

## Completion gate
This stage is complete only when the future KR project can answer:
- when should deep research be used
- how many review passes should occur for important decisions
- how should critique be structured
- when is a decision considered hardened
- how is unresolved uncertainty tracked

## Failure mode
If this stage is weak, the project will either underuse Pro’s strongest capabilities or waste them indiscriminately.

---

# Stage 5 — KR aspect system

## Purpose
This stage defines the complete map of major KR aspects that the real KR project will harden over time.

This is critical because your production model depends on long-horizon, one-aspect-at-a-time hardening.

The aspect system is what gives that approach structure.

## Required outputs
- master aspect map
- aspect hierarchy if needed
- aspect boundary rules
- criteria for when two topics belong in one aspect versus separate aspects
- aspect priority rules
- aspect state model
- criteria for when an aspect is:
  - unexplored
  - under exploration
  - provisionally hardened
  - hardened
  - blocked
  - rejected
- aspect completion rules
- cross-aspect dependency tracking method

## Completion gate
This stage is complete only when the future KR project has a coherent, non-chaotic map of what it is supposed to harden.

## Failure mode
If this stage is weak, the project will scatter into arbitrary chat topics instead of systematic coverage.

---

# Stage 6 — Repo artifact system

## Purpose
This stage defines how insights become durable repo structure.

You explicitly want the model to notice when files should exist and to convert important thinking into repo-native artifacts.

This stage makes that behavior systematic.

## Required outputs
- artifact taxonomy
- doctrine file types
- registry file types
- YAML or structured-file candidates
- decision-log model
- file proposal format
- path proposal format
- when a thought deserves a file
- when a file deserves a registry instead of prose
- artifact naming conventions
- artifact lifecycle rules
- distinction between control-plane files and substantive strategy/product files
- distinction between temporary notes and durable doctrine

## Completion gate
This stage is complete only when the future KR project can systematically convert major conclusions into well-scoped repo artifacts.

## Failure mode
If this stage is weak, valuable thinking will remain trapped in chats or produce messy repo sprawl.

---

# Stage 7 — Execution handoff system

## Purpose
This stage defines how strategic thinking is translated into bounded work for execution agents such as Codex.

Without this stage, the project may think well but execute poorly.

## Required outputs
- task-packet template
- implementation-brief template
- acceptance-criteria template
- repo-edit proposal format
- dependency declaration format
- uncertainty notes for execution workers
- review expectations for returned work
- distinction between tasks suitable for Codex and tasks unsuitable for Codex
- rule for when a task must be further hardened before being handed off

## Completion gate
This stage is complete only when the future KR project can emit execution-ready, bounded, reviewable tasks.

## Failure mode
If this stage is weak, implementation workers will receive fuzzy or strategically unsafe tasks.

---

# Stage 8 — Production-project bootstrap package

## Purpose
This stage assembles the complete startup package for the real KR project.

By this point, the logic should already exist. This stage packages it into a direct deployment sequence.

## Required outputs
- final list of files to upload to the real KR project
- upload order
- production project name recommendation
- project-memory recommendation
- final project instructions
- final global custom instructions
- pseudo-skill file set if needed
- first kickoff prompt
- first aspect recommendation
- first-chat startup procedure
- initial file tree for the project support documents

## Completion gate
This stage is complete only when the user could create the real KR project immediately by following the package.

## Failure mode
If this stage is weak, the pre-work project produces good ideas but not an actual deployable setup.

---

# Stage 9 — Pre-launch validation

## Purpose
This stage tests the setup package before the real KR project is created.

The aim is to catch hidden flaws in the operating system itself.

## Required outputs
- validation checklist
- dry-run simulation of the first production chat
- test of whether the kickoff prompt produces the intended behavior
- test of whether the model asks too many unnecessary questions
- test of whether the model keeps chats narrow enough
- test of whether durable outputs are produced cleanly
- final corrections list
- explicit launch recommendation or launch hold

## Completion gate
This stage is complete only when the operating system survives realistic validation and the pre-work project can explicitly declare the production project ready to launch.

## Failure mode
If this stage is skipped, the real KR project becomes the testing ground for setup flaws that should have been eliminated earlier.

---

## 6. No-skip rules

The following stages may not be skipped:
- Stage 1
- Stage 3
- Stage 4
- Stage 5
- Stage 8
- Stage 9

These are structurally mandatory.

If time is constrained, some stages may be compressed, but none of the above may be omitted.

---

## 7. Output rule for each stage

A stage is not complete because it was discussed.

A stage is complete only when its durable outputs exist in written form and are good enough to be transferred directly into the future real KR project.

Whenever possible, outputs should be:
- markdown files
- instruction blocks
- templates
- protocol documents
- checklists
- file trees
- startup procedures

---

## 8. Chat-level execution rule

Within the pre-work project, each stage should usually be broken into one or more dedicated chats.

Each chat should focus on one setup aspect only.

Within each chat, work should proceed in bounded subtasks.

The user may advance the work with prompts like:
- continue
- go deeper
- critique this
- harden this further
- simulate failure modes
- convert this into a file
- test this against edge cases

This is the intended rhythm.

---

## 9. Definition of “fully ready”

The pre-work project is fully ready to hand off into the real KR project only when:

1. the future model role is fully defined
2. the future instruction system is fully defined
3. the future chat workflow is fully defined
4. the future deep-research and review system is fully defined
5. the future aspect map is fully defined
6. the future repo artifact system is fully defined
7. the future execution handoff format is fully defined
8. the production bootstrap package is complete
9. the whole system has been validated in a realistic dry run

Anything less is still setup in progress.

---

## 10. Final note

This roadmap exists to create a future KR project that can begin immediately in a disciplined, high-rigor, strategically ambitious mode.

The pre-work project should therefore optimize for:
- clarity
- transferability
- rigor
- autonomy
- robustness
- maximal leverage of ChatGPT Pro’s strongest capabilities

The real project should inherit a system, not merely a set of conversations.

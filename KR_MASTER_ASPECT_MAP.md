# KR Master Aspect Map

## 1. Purpose

This document defines the master aspect map for KR.

Its purpose is to provide the future KR production project with a complete, structured, long-horizon map of the major aspects that must be hardened over time in order for KR to become a genuinely powerful, trustworthy, strategically coherent intelligent Islamic knowledge library.

This map exists to solve a central operational problem:

The future KR production project will work one aspect per chat, over a potentially very long period. Without a clear aspect system, that work will become arbitrary, unbalanced, and strategically incomplete.

The aspect map ensures that:
- important areas are not forgotten
- adjacent topics are not constantly mixed together
- one-aspect-per-chat remains practical
- hardening progress can be tracked coherently
- the production project can move from current-state bottlenecks to long-horizon end-state design without chaos

---

## 2. What an aspect is

An aspect is a major, strategically meaningful dimension of KR that is large enough to deserve repeated hardening over time and stable enough to be treated as a reusable planning category.

An aspect is not:
- a random topic
- a single implementation task
- a temporary blocker
- a vague theme
- a one-off question

An aspect is something like:
- input-format strategy
- excerpting quality doctrine
- taxonomy architecture
- provenance and attribution model
- retrieval and study workflows
- evaluation doctrine
- autonomy and execution policy

Each aspect should be substantial enough to support multiple dedicated chats over time.

---

## 3. Aspect design goals

The KR aspect system should satisfy all of the following:

1. **Coverage**
   The map should cover the major strategic dimensions of KR, not only the currently active pipeline stage.

2. **Non-chaos**
   The map should reduce arbitrary topic drift.

3. **Separation**
   Aspects should be distinct enough that one-aspect-per-chat is workable.

4. **Scalability**
   The map should support long-horizon growth without needing total redesign every few weeks.

5. **Strategic relevance**
   Each aspect should matter to KR’s real future, not just be administratively neat.

6. **Expandable structure**
   The map should support subaspects and derived questions without collapsing into disorder.

---

## 4. Structural levels

The aspect system should operate at three levels.

### Level A — domains
Broad top-level families of concerns.

### Level B — aspects
The main enduring units of hardening work.
These are the primary one-chat-per-aspect targets.

### Level C — subaspects
Important sub-questions nested under an aspect.
These may sometimes deserve their own dedicated chats, but they remain subordinate to the parent aspect.

The production project should usually anchor chats at Level B.
Level C is used when the sub-question becomes large enough to deserve its own thread.

---

## 5. Top-level domain map

The major KR domains are:

1. Mission and end-state
2. User and study workflows
3. Corpus and input acquisition
4. Knowledge transformation pipeline
5. Knowledge representation and library structure
6. Trustworthiness and scholarly integrity
7. Retrieval, interaction, and product behavior
8. Evaluation and quality control
9. Repo structure and durable artifacts
10. Agent workflow and execution system

These domains are not the default chat units.
They are the top-level grouping system.

The default chat units are the aspects inside them.

## 7. Aspect priority classes

Not all aspects are equally urgent at all times.

Each aspect should be classified into one of the following priority classes relative to the current project stage.

### P0 — immediate bottleneck
Directly blocks the project’s most important current progress.

### P1 — near-term enabling
Not the bottleneck itself, but materially enables the next stage of good work.

### P2 — strategic foundation
Not immediately urgent, but important enough that future work will be weaker without it.

### P3 — later-state expansion
Valuable, but should not yet dominate the project’s attention.

Priority is stage-dependent.
The same aspect may change class over time.

Each aspect should be tracked through a state model.

Suggested states:

- unentered
- scoped
- under exploration
- mapped
- provisionally directed
- under critique
- provisionally hardened
- hardened
- blocked by evidence
- blocked by user decision
- deferred
- rejected

This state model helps the production project know not only what aspects exist, but where each one currently stands.

## 9. Aspect boundary rules

To preserve one-aspect-per-chat discipline, the production project should use the following boundary rules.

### Rule 1 — Parent aspect first
A sub-question belongs to the parent aspect unless it becomes large enough to require its own dedicated chat.

### Rule 2 — No silent migration
If a chat drifts from one aspect to another, that shift should be recognized explicitly.

### Rule 3 — Shared edges do not erase boundaries
Two aspects may be tightly related without becoming the same aspect.

Example:
- input-format strategy and source acquisition are related, but not identical
- taxonomy doctrine and retrieval doctrine are related, but not identical
- synthesis doctrine and interaction model are related, but not identical

### Rule 4 — Aspect first, implementation later
A strategic aspect should not collapse into implementation details too early.

### Rule 5 — Local question must serve the aspect
A sub-question is worth pursuing only if resolving it materially hardens the parent aspect.

---

## 10. Criteria for splitting a subaspect into its own chat

A subaspect deserves its own chat when one or more of the following are true:

- it has a clearly different objective than the parent thread
- it requires a different kind of reasoning or evidence
- it could generate its own durable artifact
- it introduces enough detail that the parent chat would lose clarity
- it is likely to require multiple turns of hardening on its own

Examples:
- inside input-format strategy, “PDF support doctrine” may become its own chat
- inside retrieval doctrine, “taxonomy-driven browse versus lexical search” may become its own chat
- inside synthesis doctrine, “citation-grounded synthesis boundaries” may become its own chat

---

## 11. Criteria for rejecting a supposed aspect

Not every recurring topic deserves aspect status.

A topic should not become a formal aspect if it is:
- too small
- too temporary
- purely implementation-local
- already fully contained by another aspect
- unlikely to recur
- administratively neat but strategically unimportant

The map should remain disciplined.
Too many aspects destroy usefulness.

## 13. Coverage warning

The project must not confuse “pipeline coverage” with “system coverage.”

KR is not only:
- source
- normalization
- excerpting
- taxonomy
- synthesis

Those are central transformation stages, but they do not cover:
- user value
- trustworthiness doctrine
- provenance
- retrieval
- library structure
- evaluation
- repo artifact systems
- autonomy and execution policy

The master aspect map exists partly to prevent that narrowing failure.

## 14. Anti-failure rules

The production project must avoid the following aspect-system failures.

### 14.1 Bottleneck myopia
Only hardening what is immediately blocking code, while neglecting end-state, value, and trust questions.

### 14.2 Feature-sprawl mapping
Turning every idea into a new aspect.

### 14.3 Pipeline reductionism
Treating KR as only a pipeline rather than a library system.

### 14.4 Product detachment
Developing product aspects without grounding them in source integrity and scholarly seriousness.

### 14.5 Doctrine vacuum
Working on many local questions without formalizing the aspect-level doctrine.

### 14.6 Static-map failure
Treating the aspect map as frozen and untouchable forever.

### 14.7 Map-churn failure
Rewriting the aspect map too frequently without strong reason.

The map should be stable, but revisable deliberately.

---

## 15. Maintenance rule

The master aspect map should not be edited casually.

It should be revised only when:
- KR’s scope materially changes
- a major missing aspect is discovered
- two aspects are found to be wrongly separated or wrongly merged
- the system needs a clearer hierarchy
- the long-horizon understanding of KR becomes sharper

Routine chat outcomes should not constantly mutate the map.

The aspect map is meant to be a durable navigation system.

---

## 16. Success condition

The KR master aspect map is successful only if the future KR production project can use it to do all of the following:

- choose meaningful next aspects to harden
- keep one-aspect-per-chat discipline
- avoid major blind spots
- distinguish immediate bottlenecks from deeper foundations
- understand how local questions fit the wider system
- convert repeated discussion into systematic coverage
- grow over many chats without losing coherence

That is the intended function of the map.

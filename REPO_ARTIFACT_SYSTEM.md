# KR Repo Artifact System

## 1. Purpose

This document defines how the future KR production project should convert important thinking into durable repo-native artifacts.

Its purpose is to ensure that:
- major conclusions do not remain trapped in chat
- the repo accumulates usable strategic structure rather than documentation clutter
- file creation is deliberate rather than ad hoc
- the right kinds of artifacts exist for the right kinds of project logic
- KR’s long-horizon operating intelligence becomes recoverable from the repo

This system is a bridge between:
- reasoning
- doctrine
- project control
- execution
- durable memory

It exists because an intelligent project cannot depend on conversational memory alone.

---

## 2. Core principle

The repo is not merely where implementation code lives.

For KR, the repo must also contain the durable strategic structure that allows the project to:
- preserve important reasoning
- guide future work
- track commitments
- define doctrine
- expose evaluation logic
- structure execution
- remain recoverable across sessions, tools, and agents

However, not every good thought deserves a file.

The repo-artifact system must therefore optimize for both:
- preservation of high-value structure
- resistance to document sprawl

---

## 3. Governing law

A conclusion should become a repo artifact only when doing so materially improves one or more of the following:

- future decision quality
- future execution quality
- recoverability of project state
- coherence across sessions
- consistency of standards
- reusability of methods
- visibility of commitments
- auditability of reasoning

If none of these improve materially, the conclusion should usually remain in chat or be folded into an existing artifact instead of spawning a new file.

---

## 4. Core artifact classes

The future KR production project should think in terms of artifact classes.

Each class solves a different problem and should not be confused with the others.

### 4.1 Control-plane artifacts
Purpose:
- govern project state and live direction

Examples:
- charter
- active frontier
- handoff
- decisions ledger

Characteristics:
- high authority
- tightly scoped
- updated by rule
- should remain minimal and clean

These belong in the control plane and must not absorb all other project logic.

---

### 4.2 Doctrine artifacts
Purpose:
- state durable project positions on important strategic questions

Examples:
- end-state doctrine
- v1 proof doctrine
- taxonomy doctrine
- trustworthiness doctrine
- input-format doctrine

Characteristics:
- durable
- conceptual
- reused across many future decisions
- should capture stable project thinking rather than transient debate

Doctrine artifacts answer:
- what KR believes
- why
- where the boundary lies
- what principles govern future decisions

---

### 4.3 Protocol artifacts
Purpose:
- define how recurring work should be done

Examples:
- review protocol
- deep-research protocol
- chat workflow protocol
- handoff protocol
- artifact proposal protocol

Characteristics:
- procedural
- reusable
- method-oriented
- should reduce repeated rethinking of process

Protocol artifacts answer:
- how KR operates
- how certain recurring tasks should be handled

---

### 4.4 Architecture artifacts
Purpose:
- define how major system structures are organized

Examples:
- representation architecture notes
- library structure design
- pipeline coherence memos
- intelligence interaction model
- autonomy boundary architecture

Characteristics:
- structural
- design-oriented
- often connect multiple aspects
- should focus on shape and relation, not just opinion

Architecture artifacts answer:
- how KR is structured
- how major parts fit together

---

### 4.5 Evaluation artifacts
Purpose:
- define how KR judges quality, readiness, and failure

Examples:
- evaluation doctrine
- acceptance criteria frameworks
- benchmark selection notes
- calibration rules
- error taxonomy
- review rubrics

Characteristics:
- normative
- operational
- often used to gate execution or accept outputs

Evaluation artifacts answer:
- how KR knows whether something is good enough

---

### 4.6 Registry artifacts
Purpose:
- store structured, updateable, reference-like project inventories

Examples:
- aspect registry
- doctrine registry
- artifact registry
- unresolved-question registry
- file inventory
- benchmark registry

Characteristics:
- structured
- compact
- reference-oriented
- often better suited to tables, YAML, or other structured formats

Registry artifacts answer:
- what exists
- what state it is in
- what remains open
- what belongs where

---

### 4.7 Schema artifacts
Purpose:
- define structured shapes that future files or systems should follow

Examples:
- decision record schema
- task packet schema
- unit-of-knowledge schema
- artifact proposal schema
- YAML structure definitions

Characteristics:
- format-defining
- highly reusable
- should reduce ambiguity in repeated artifact creation

Schema artifacts answer:
- what fields or sections are required
- what structure a recurring artifact should take

---

### 4.8 Template artifacts
Purpose:
- provide repeatable starting forms for recurring deliverables

Examples:
- doctrine note template
- decision record template
- Codex task packet template
- review memo template
- aspect chat opener template

Characteristics:
- execution-supportive
- reusable
- should make quality easier to reproduce

Template artifacts answer:
- how to instantiate a recurring artifact quickly and well

---

### 4.9 Working artifacts
Purpose:
- hold temporary but still useful structured intermediate thinking

Examples:
- scoped exploration notes
- comparison memos
- pre-decision drafts
- staged conversion notes

Characteristics:
- temporary or semi-durable
- should not be allowed to accumulate without lifecycle discipline
- may later be promoted, merged, or deleted

Working artifacts answer:
- what is currently being explored before it is mature enough to become doctrine or architecture

---

### 4.10 Execution artifacts
Purpose:
- drive implementation or bounded action

Examples:
- task packets
- implementation briefs
- file-creation tickets
- validation checklists
- review requests

Characteristics:
- action-oriented
- should be tightly scoped
- must connect directly to hardened strategic reasoning

Execution artifacts answer:
- what exactly should now be done

---

## 5. Artifact-type selection logic

When a durable concept emerges, the future KR production project should ask:

### Question 1
Is this primarily about governing the live project state?
If yes, it may belong in the control plane.

### Question 2
Is this primarily a durable project belief or boundary?
If yes, it is likely a doctrine artifact.

### Question 3
Is this primarily a reusable method?
If yes, it is likely a protocol artifact.

### Question 4
Is this primarily about system shape or major structural relationships?
If yes, it is likely an architecture artifact.

### Question 5
Is this primarily about how quality is judged?
If yes, it is likely an evaluation artifact.

### Question 6
Is this primarily a structured inventory or status set?
If yes, it is likely a registry artifact.

### Question 7
Is this primarily defining a format or required fields?
If yes, it is likely a schema artifact.

### Question 8
Is this primarily a reusable starting form?
If yes, it is likely a template artifact.

### Question 9
Is this not mature enough to become durable doctrine, but still too useful to lose?
If yes, it may temporarily be a working artifact.

### Question 10
Is this now ready to drive implementation or bounded execution?
If yes, it is likely an execution artifact.

This selection logic must be used deliberately.
The same idea should not become several redundant artifact types without reason.

---

## 6. When a thought deserves a file

A thought deserves a file when one or more of the following are true:

### 6.1 Reuse test
It will likely be needed repeatedly in future work.

### 6.2 Governing test
It should constrain future decisions or workflows.

### 6.3 Recoverability test
Losing it to chat history would materially harm future progress.

### 6.4 Coordination test
It helps multiple future agents or sessions work consistently.

### 6.5 Execution test
It is needed to convert strategic reasoning into action.

### 6.6 Auditability test
It records a commitment, standard, boundary, or rationale that should remain inspectable.

### 6.7 Structure test
It is large or important enough that keeping it implicit would create confusion.

If none of these tests are met, a new file is usually unnecessary.

---

## 7. When a thought does NOT deserve a new file

A thought should usually not become a new file if it is:

- still highly unstable and likely to change substantially soon
- too small to justify durable storage
- already adequately captured in an existing file
- only relevant to a single ephemeral local turn
- easily restatable when needed
- just a conversational formulation rather than a real project commitment
- more appropriate as an update to a registry or existing doctrine note

This rule is critical to prevent file proliferation.

---

## 8. Update-existing versus create-new rule

The default question should not be:
- “Should I create a new file?”

It should be:
- “Does this belong in an existing durable home, or is a new artifact truly justified?”

### Create a new artifact when:
- the concept has distinct purpose and scope
- no existing file is the right home
- merging would reduce clarity
- the concept is durable enough to stand on its own

### Update an existing artifact when:
- the concept is a refinement of an existing doctrine
- the concept is another entry in an existing registry
- the concept belongs naturally under an existing protocol
- a separate file would create fragmentation or duplication

The repo should evolve by disciplined accretion, not file explosion.

---

## 9. Control-plane boundary law

The control plane must remain clean.

This means:
- `.kr/` should contain project-control artifacts only
- `.kr/` should not become the dump site for all strategic thought
- substantive doctrine, architecture, evaluation, and artifact systems should live outside the control plane in their own appropriate locations
- control-plane files should point, govern, and summarize, not absorb everything

The future production project should treat `.kr/` as:
- constitutional and live-directional
not as
- the total documentation namespace

This distinction is non-negotiable.

---

## 10. Recommended top-level strategic artifact families

Outside the control plane, the future KR production project should think in terms of strategic artifact families such as:

- vision/
- doctrine/
- protocols/
- architecture/
- evaluation/
- registries/
- templates/
- execution/

These names are placeholders for the pre-work project to refine later if needed, but the conceptual separation should remain.

The goal is to make the repo legible by artifact function.

---

## 11. Prose versus structured-data rule

Not every artifact should be prose.

The future KR production project should choose between prose and structured data deliberately.

### Prefer prose when:
- nuance matters
- reasoning boundaries matter
- tradeoffs must be explained
- conceptual doctrine is being established
- interpretation is central

Examples:
- doctrine notes
- architecture memos
- strategic recommendations
- trustworthiness boundaries

### Prefer structured data when:
- repeatable fields matter more than narrative explanation
- multiple entries of the same kind will exist
- machine readability or consistency would help
- status tracking is important
- inventories and references dominate the use case

Examples:
- registries
- state maps
- decision indexes
- artifact inventories
- unresolved-question trackers

### Prefer hybrid structure when:
- both explanation and structured fields are needed

Examples:
- decision records
- task packets
- review memos
- acceptance dossiers

The future project should avoid forcing everything into markdown prose when a registry or YAML would be better.

---

## 12. When to use YAML or another structured file

YAML or equivalent structured formats should be used when all of the following are substantially true:

- the artifact has a repeated schema
- entries are easier to compare when normalized
- future automation or validation may benefit
- prose would add unnecessary ambiguity
- the artifact is primarily reference/state rather than argument

Good candidates:
- aspect registry
- artifact registry
- unresolved-question registry
- file metadata manifests
- task packet manifests
- benchmark inventories

Poor candidates:
- nuanced doctrine notes
- architectural arguments
- conceptual boundary documents
- long-form recommendations

The future project must not use YAML merely because it feels technical.
It should use it when structure is genuinely the dominant need.

---

## 13. Artifact maturity ladder

Important concepts should move through a maturity ladder.

### M0 — chat-only
Not yet durable enough to formalize.

### M1 — working form
Useful intermediate note or comparison memo exists.

### M2 — durable draft
The concept is stable enough to deserve a real artifact draft.

### M3 — accepted artifact
The artifact is mature enough to guide future work.

### M4 — governing artifact
The artifact now plays a meaningful role in constraining later decisions or execution.

This maturity model helps avoid both:
- premature formalization
- and endless non-conversion

---

## 14. Artifact proposal protocol

Whenever the future KR production project concludes that a durable artifact should exist, it should not stop at “we should make a file.”

It should propose the artifact in a structured form such as:

### Artifact proposal format

- proposed path:
- proposed file name:
- artifact class:
- purpose:
- why this should exist:
- why this should be a separate artifact instead of an update to an existing one:
- recommended format:
- expected sections or schema:
- maturity level:
- whether it is immediately needed or later:
- relationship to existing files:
- whether it is doctrine, protocol, registry, schema, template, or execution support:
- exact next move:

This format is essential.
It turns vague repo suggestions into actionable structure.

---

## 15. Artifact naming principles

Names should optimize for:
- clarity
- scope legibility
- durable usefulness
- grepability
- low ambiguity

Names should avoid:
- vague labels like `notes.md`, `ideas.md`, `misc.md`
- timestamps unless the artifact is inherently time-bound
- private conversational wording
- cute names that hide function

Good naming patterns tend to express:
- what the artifact governs
- what class it is
- what its role is

Examples:
- `INPUT_FORMAT_DOCTRINE.md`
- `TRUSTWORTHINESS_DOCTRINE.md`
- `ASPECT_REGISTRY.yaml`
- `CODEX_TASK_PACKET_TEMPLATE.md`

This principle matters more than any one exact naming style.

---

## 16. Artifact lifecycle rules

Every durable artifact should have an implicit or explicit lifecycle.

Possible lifecycle states:
- proposed
- drafted
- active
- superseded
- deprecated
- archived

The future KR production project should avoid “zombie artifacts” that still sit in the repo but no longer govern anything.

When an artifact is superseded:
- the replacement should be clear
- the status of the old one should be explicit
- future readers should not have to guess which file is authoritative

Lifecycle discipline is part of repo trustworthiness.

---

## 17. Registry doctrine

Registries are especially important because the future KR production project will manage many recurring structures:
- aspects
- doctrines
- unresolved questions
- artifacts
- benchmarks
- possibly tasks and execution states

A registry should be used when:
- multiple entries of the same type will exist
- consistency matters
- cross-reference value is high
- a living inventory is more useful than repeated prose

A registry should not become a substitute for doctrine.
It stores structured state; it does not carry all conceptual reasoning.

---

## 18. Doctrine doctrine

A doctrine artifact should exist when the project needs a durable answer to a question of the form:

- what is KR’s position on X?
- what is the correct boundary around X?
- what principles govern future decisions about X?
- what counts as acceptable or unacceptable within X?

Doctrine artifacts should:
- be conceptually stable
- include boundaries
- include rationale
- define implications
- distinguish current state, v1, and long-horizon vision when relevant

Doctrine artifacts should not become implementation scratchpads.

---

## 19. Protocol doctrine

A protocol artifact should exist when the project needs a repeatable answer to a question of the form:

- how should this recurring activity be done?
- what steps should be followed each time?
- what review sequence or checklist applies?
- what output format is expected?

Protocols should:
- reduce repeated rethinking
- define sequence and constraints
- specify expected outputs
- state failure modes where relevant

Protocols should be reusable methods, not one-off memos.

---

## 20. Execution-artifact doctrine

An execution artifact should exist when strategic thinking is mature enough to drive bounded work.

An execution artifact should:
- define the task
- define why the task matters
- define dependencies
- define acceptance criteria
- define constraints
- define what is out of scope
- define review expectations

Execution artifacts are not strategic theory.
They are bounded action instruments.

The future KR production project should not hand work to Codex or any implementation agent without the right execution artifact shape.

---

## 21. Artifact review criteria

Before accepting a proposed artifact, the future project should test it against the following questions:

### 21.1 Necessity
Does this artifact materially improve the repo?

### 21.2 Non-duplication
Is this duplicating something that already exists?

### 21.3 Correct class
Is this the right artifact type?

### 21.4 Correct format
Should this be prose, structured data, template, or schema?

### 21.5 Clear authority
Will future readers know what this artifact is for?

### 21.6 Good placement
Does it belong where it is being placed?

### 21.7 Reusability
Will this actually help future work?

### 21.8 Lifecycle plausibility
Will this artifact have a sensible durable life?

If these criteria are not met, the artifact should be revised, merged, deferred, or rejected.

---

## 22. Anti-failure rules

The future KR production project must avoid the following artifact-system failures.

### 22.1 Chat-trap failure
Important conclusions remain trapped in chats and never become durable artifacts.

### 22.2 File-sprawl failure
Every interesting thought becomes a new file.

### 22.3 Wrong-format failure
The project uses prose when a registry or schema is needed, or uses YAML when nuance is central.

### 22.4 Control-plane pollution failure
The control plane absorbs doctrine, architecture, and unrelated strategic content.

### 22.5 Zombie-file failure
Old files remain without clear authority or lifecycle state.

### 22.6 Naming-chaos failure
Files are named vaguely or inconsistently.

### 22.7 Redundancy failure
The same idea is copied across multiple files without authority clarity.

### 22.8 Premature-formalization failure
The project creates durable files for unstable thoughts too early.

These failures must be actively resisted.

---

## 23. Conversion rule from aspect hardening

When an aspect chat reaches a meaningful local conclusion, the model should ask:

- does this conclusion deserve doctrine?
- does it require a protocol?
- should it update a registry?
- does it imply a schema or template?
- is it ready to become an execution artifact?
- or should it remain chat-local for now?

This question should be routine.
It is one of the main mechanisms by which the future production project converts intelligence into repo structure.

---

## 24. Success condition

The repo artifact system is successful only if the future KR production project behaves like this:

- it notices when durable artifacts should exist
- it chooses the right artifact class
- it chooses the right format
- it keeps the control plane clean
- it avoids document sprawl
- it preserves important reasoning in recoverable form
- it makes future work easier and more coherent
- it enables high-quality execution and review

That is the intended standard.

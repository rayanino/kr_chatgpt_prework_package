# KR Production Bootstrap Protocol

## 1. Purpose

This document defines the exact procedure for creating and launching the real KR production project after the pre-work project is complete.

Its purpose is to ensure that the production project does not begin in a partially configured, ambiguous, or contaminated state.

The production project should not be treated as a place to “figure out the setup as we go.”
It should begin already equipped with:
- the right behavioral law
- the right project files
- the right startup sequence
- the right first aspect
- the right operating rhythm

This protocol exists to make launch clean, immediate, and reliable.

---

## 2. Governing rule

The KR production project must be created only after the pre-work project has produced a complete deployable setup package.

The production project must not become a hybrid of:
- setup discussion
- production work
- scattered experimentation
- repeated re-teaching of behavior

The goal is:
- one clean launch
- one coherent project mind
- one disciplined startup sequence

---

## 3. What the production project is

The production project is the long-horizon working home in which ChatGPT Pro acts as:
- KR’s thinking owner
- creative product manager
- senior architect
- doctrine hardener
- systems director
- strategic reviewer

It is not the place to design how ChatGPT should behave.
That work belongs to the pre-work project.

The production project begins only after its operating system already exists.

## 6. Project memory rule

The production project should be created as a fresh project with the intended isolated memory behavior.

The goal is:
- strong internal coherence
- minimal contamination from unrelated chats
- all major KR reasoning staying inside the project

The production project should begin clean and self-contained.

Do not use an already messy multi-purpose project for KR production work.

---

## 7. Global setup sequence

The production bootstrap should happen in the following order.

### Step 1 — finalize account-level baseline
Ensure the global custom instructions are already set to the finalized baseline version.

### Step 2 — create the production project
Create a new project specifically for KR production work.

### Step 3 — set the project instructions
Paste the finalized production-project instructions immediately.

Do not begin working chats before this is done.

### Step 4 — upload the production file bundle
Upload the initial curated file set in the intended sequence.

### Step 5 — verify the project’s operating reference set
Confirm that the project contains the intended governing and support files, not random clutter.

### Step 6 — start the first real aspect chat
Use the finalized first-chat opener.
Do not begin with an improvised generic prompt.

### Step 7 — validate behavior early
Before treating the project as fully launched, verify that the first chat behaves according to the intended operating model.

This order should not be casually rearranged.

## 8. Production file bundle philosophy

The production project should not begin with every potentially relevant file.

It should begin with a disciplined, curated initial bundle containing only files that are:
- foundational
- governing
- strategically necessary
- immediately useful

Too few files creates under-context.
Too many files creates noise.

The initial bundle should optimize for:
- clarity
- authority
- reusability
- low clutter

## 11. Upload discipline rules

When uploading files into the production project:

### Rule 1
Do not upload everything available just because it exists.

### Rule 2
Prefer canonical and authoritative files over redundant variants.

### Rule 3
Prefer durable files over temporary working clutter.

### Rule 4
If two files overlap, upload the one with clearer authority.

### Rule 5
If a file is likely to distract early work more than help it, delay it.

### Rule 6
The project should begin with enough context to act strongly, not so much context that its center of gravity becomes muddy.

---

## 12. What should NOT be in the initial bundle

The production project should not initially be loaded with:
- obsolete draft files
- experimental notes with unclear authority
- old handoff fragments outside the canonical control plane
- broad backlog dumps
- duplicate versions of the same doctrine
- large volumes of low-authority brainstorming
- files unlikely to matter in the first few aspects

These can be introduced later if needed.

At launch, clarity is more valuable than completeness.

## 13. Final global custom instructions

The account-level global custom instructions should be set to the finalized baseline version before production launch.

### Recommended global custom instructions

Work with me as a rigorous strategic collaborator. Prioritize precision, explicit reasoning, and honest uncertainty. Challenge weak assumptions early. Prefer bottleneck-first thinking over feature sprawl. Distinguish clearly between facts, inferences, options, and recommendations. Optimize for long-term leverage, coherence, and downstream quality rather than speed or activity. Keep writing dense but readable. Avoid flattery, filler, and vague encouragement. When something is underdefined, surface the missing definition instead of silently assuming it.

These instructions are intentionally generic.
They are not the main production behavior layer.

## 14. Final production-project instructions

The project instructions should be pasted into the production project before any real chat begins.

### Recommended production-project instructions

This project is KR’s strategic command center.

Your role is not assistantship. Your role is to act as KR’s creative product manager, senior architect, doctrine hardener, systems director, and strategic reviewer.

Take initiative. Default to deciding rather than asking. Ask the user questions only when the answer genuinely depends on user preference, permissions, available tools/accounts, irreversible choices, or external evidence that cannot be responsibly inferred. If a reasonable assumption can be made, make it, state it explicitly when meaningful, and proceed.

Understand KR as it currently exists, but do not treat the repo’s current vision as final. Continuously search for strategically valuable expansions, missing capabilities, missing workflows, missing input/output formats, missing abstractions, and missing evaluation structures. Improve KR’s long-horizon usefulness as an intelligent Islamic knowledge library.

Work one aspect at a time. Harden each aspect from the inside out through multiple rounds of critical reasoning. Prefer small, durable advances over broad vague discussion. End serious discussions with durable outputs such as a spec, doctrine note, architecture recommendation, decision, rubric, file plan, task packet, or clearly defined unresolved question.

Use the repo as evidence and current state, not as a prison. Respect existing durable law where it serves the project. Challenge weak assumptions, narrow constraints, and missing structure. Whenever the project needs a durable artifact, propose the exact file path, file purpose, and expected content.

Lead decisively in product, architecture, workflow, evaluation, prioritization, and repo structure. For Islamic-science commitments, do not bluff. Mark uncertainty clearly, separate speculative design from asserted correctness, and create research tasks when responsible confidence is not possible.

Be direct and intellectually serious. Surface hidden assumptions. Critique weak ideas early. Optimize for long-term leverage, coherence, and trustworthiness. Minimize unnecessary questions to the user.

## 15. First-production-chat philosophy

The first production chat is not a generic “hello.”
It is the first real activation of the production operating system.

Therefore, the first chat should:
- focus on one aspect only
- produce a durable output
- validate that the project is acting in the intended role
- avoid sprawling into broad project review
- avoid repeating the setup phase

The first production chat should prove that the launch succeeded.

---

## 16. First aspect selection rule

The first aspect should be chosen using the following criteria:

### 16.1 High leverage
It should materially affect future project quality.

### 16.2 Suitable for strategic hardening
It should benefit from strong reasoning, not only implementation.

### 16.3 Narrow enough to contain
It should be possible to keep the first chat focused.

### 16.4 Broad enough to matter
It should produce a durable artifact that future work can rely on.

### 16.5 Connected to current KR reality
It should not be a random later-stage curiosity.

The first aspect should not be selected merely because it is exciting.
It should be selected because it is a strong activation target.

## 18. First production chat opener

Use a deliberate opener rather than improvising.

### Recommended first production chat opener

Aspect:
A2 — V1 proof doctrine

Objective:
This chat is for defining exactly what KR v1 must prove, what it explicitly does not need to prove, and what minimum conditions must be true for v1 to count as strategically meaningful.

Scope boundary:
Stay focused on v1 proof doctrine only. Do not widen into the full long-horizon roadmap unless necessary to clarify the boundary between v1 and later-state ambition.

Desired output:
A durable doctrine note defining KR v1 proof scope, non-goals, minimum proof requirements, and implications for future evaluation and execution.

Special constraints:
Anchor yourself in the current repo state and control plane, but do not treat current assumptions as final. Work one bounded subtask at a time. Minimize questions to the user. Convert the result into a durable artifact proposal when ready.

Working rule:
Lead the reasoning. Keep the chat narrow. Use strong defaults where reasonable. Surface uncertainty explicitly. Critique weak assumptions. Aim for a durable result, not broad discussion.

## 19. Expected first-chat output

The first production chat should ideally produce one or more of:

- a draft `V1_PROOF_DOCTRINE.md`
- a clear definition of what v1 must prove
- a list of explicit non-goals
- a first-cut minimum acceptance frame for v1
- identified unresolved questions that deserve follow-up chats
- a proposal for any additional file(s) that should exist because of this doctrine

The goal is not to finish all of KR in one chat.
The goal is to prove that the project is operationally alive.

## 20. Early-launch validation checklist

After the first production chat, validate whether the project behaved correctly.

### Behavior validation
- Did it take the lead rather than wait passively?
- Did it minimize unnecessary questions?
- Did it stay within one aspect?
- Did it work one bounded subtask at a time?
- Did it challenge weak assumptions?
- Did it produce durable outputs or clear artifact proposals?

### Scope validation
- Did it avoid broadening into multiple unrelated aspects?
- Did it remain anchored in current KR reality?
- Did it avoid acting as a mere repo summarizer?

### Hardening validation
- Did it critique the emerging direction?
- Did it distinguish between current state, v1, and later-state ambition?
- Did it identify unresolved questions rather than faking closure?

### Operational validation
- Did it feel like a thinking owner rather than a generic assistant?
- Did the “one aspect, one subtask at a time” rhythm work?
- Could the user plausibly continue the thread with a simple “continue”?

If the answer to several of these is no, the production setup likely needs correction.

---

## 21. Launch failure modes

The following are signs that production launch has gone badly:

### 21.1 Passive-assistant launch
The project asks too many questions and waits to be led.

### 21.2 Setup-regression launch
The project behaves as if the setup was never done and starts re-deriving the operating system.

### 21.3 Broad-review launch
The first chat becomes an unfocused tour of the whole project.

### 21.4 Repo-servant launch
The project treats the existing repo as unquestionable final truth.

### 21.5 No-artifact launch
The project talks intelligently but does not produce durable outputs.

### 21.6 Scope-instability launch
The first chat cannot stay within one aspect.

### 21.7 Overconfident-domain launch
The project makes domain-sensitive claims too casually.

If these failures appear, the launch should be corrected early rather than tolerated.

---

## 22. Post-launch correction rule

If the first production chat reveals setup weakness, do not let the production project become the place where setup keeps being improvised indefinitely.

Instead:
- identify the specific setup failure
- correct the relevant instruction, protocol, or file
- update the production project’s support system deliberately
- then continue with a clean corrected operating state

The production project may need occasional setup maintenance, but it must not relapse into being a second pre-work project.

---

## 23. Transition rule from pre-work to production

The transition from pre-work to production should be explicit.

A clean transition means:
- pre-work outputs are considered transferred
- production behavior is considered live
- future work happens inside the production project unless there is a special reason not to
- the pre-work project becomes reference material, not the active worksite

This prevents meta-work from leaking into production endlessly.

---

## 24. Production launch completion condition

The production project should be considered successfully launched only when:

- the project exists with the finalized instructions installed
- the initial upload bundle is in place
- the first aspect chat has been run
- the behavior validation checklist mostly passes
- the first chat produced durable value
- the project feels operationally self-contained
- the user can now continue with disciplined aspect-by-aspect work rather than further foundational setup

Until then, launch is still in progress.

---

## 25. Success condition

This bootstrap protocol is successful only if the real KR production project can be created and started with:
- no ambiguity about setup order
- no ambiguity about what to upload
- no ambiguity about what to paste
- no ambiguity about how to start the first chat
- no ambiguity about what good launch behavior looks like

The goal is not to create a project eventually.
The goal is to make launch procedural, clean, and repeatable.

# KR Deep Research Request Template

## 1. Purpose

This template defines the standard form the future KR production project should use when formulating a deep-research request.

Its purpose is to ensure that research is:
- sharply targeted
- tied to a real project decision
- explicit about what uncertainty needs reduction
- efficient rather than decorative
- returned in a form that can feed directly into KR’s strategic hardening

This template is not for casual browsing.
It is for evidence-gated questions that materially affect KR’s design, doctrine, architecture, sequencing, or execution safety.

---

## 2. When to use this template

Use this template when:
- the question is genuinely evidence-gated
- internal reasoning has already clarified the real unknown
- the answer would materially affect KR’s direction, quality, or execution
- the project knows what kind of evidence would actually help

Do not use this template when:
- the issue is mainly internal design
- the issue is mainly about clarifying concepts
- the issue can be resolved by critique rather than research
- the request is too vague to guide useful evidence gathering

If the request cannot be filled honestly, the research need is not yet properly framed.

---

## 3. Core principle

A research request must always serve a decision, boundary, or unresolved question.

Research is not performed:
- to look thorough
- to collect facts for their own sake
- to postpone judgment
- to replace internal reasoning

Research is performed because the project has reached a specific evidence boundary.

The request should therefore always make clear:
- what needs to be known
- why it matters
- what decision it informs
- what would count as sufficient evidence
- what kind of return would be useful

5. Section-by-section guidance
Research title

Use a short operational label.

Good:

PDF support strategic necessity
Current OCR quality constraints for Arabic scholarly PDFs
Taxonomy legitimacy constraints in classical Islamic-science classification

Bad:

Research on PDFs
Check some things
Investigate this topic

The title should make later retrieval easier.

Research trigger class

This should say why research is needed.

Good trigger classes include:

evidence
scholarly
comparative-landscape
tool-capability
corpus-reality
standards

This field helps the project remember what kind of epistemic boundary it hit.

Core research question

This is the most important field.

It should be:

one real question
decision-relevant
narrow enough to answer
broad enough to matter

Good:
Is PDF support strategically necessary for a serious long-horizon Islamic library, given likely corpus acquisition realities and the limitations of Shamela-only ingestion?

Bad:
Tell me about PDF support.

The research question must be shaped for resolution, not exploration for its own sake.

Why this matters for KR

This section must connect the research to project value.

Good:
If KR remains effectively limited to Shamela exports, the long-term corpus ceiling may be too narrow for the intended library. This question materially affects input-format doctrine and future pipeline architecture.

Bad:
This is interesting and useful.

The request should make the stakes explicit.

Current project context

This section gives enough local grounding to avoid shallow research.

Examples:

relevant aspect ID
current doctrine state
current unresolved question
current repo limitation
current stage of KR

This is where the research request ties itself back into KR’s actual operating system.

What is already believed internally

This section is important because it prevents research from pretending the project has no prior reasoning.

Good:
The current internal leaning is that PDF support is probably strategically important, but the project does not yet know whether it belongs in v1, medium-term architecture, or later-stage expansion.

Bad:
We have no idea.

Usually the project does have some current judgment. It should be stated honestly but provisionally.

What is genuinely unknown

This field is crucial.

It should say what the project truly does not know yet.

Good:
The project does not know how frequently strategically valuable Islamic works are realistically available only or primarily as PDF, how difficult high-quality extraction is in practice for Arabic scholarly texts, and whether delaying PDF support would materially cap corpus ambition.

Bad:
Everything about this topic is unknown.

That is too unfocused.

Decision or artifact this research informs

This field prevents research drift.

Good:
This research informs:

A8 input-format strategy
the future INPUT_FORMAT_DOCTRINE.md
sequencing of ingestion-related roadmap decisions

Bad:
This may be useful later.

The project should know what the research will change if it is successful.

Evidence needed

This field defines what kinds of evidence would actually help.

Examples:

evidence about corpus-format realities
evidence about current OCR/extraction capabilities
evidence about typical availability of Islamic texts in structured versus PDF formats
evidence about cost/complexity implications

This should not be a giant evidence wish list.
Only include what is decision-relevant.

Evidence that would NOT be sufficient

This is an important anti-noise field.

Examples:

one anecdotal blog post
generic claims about PDFs being “common”
marketing material from OCR vendors without practical relevance
tool descriptions that do not address Arabic scholarly text realities

This field protects the project from weak evidence looking stronger than it is.

Key comparison axes

Use this when the research is comparative.

Examples:

coverage gain
extraction difficulty
architectural complexity
trustworthiness implications
timing relevance for v1 versus later-state
cost of delay

These axes help make the returned synthesis more useful.

Constraints

This section keeps research properly bounded.

Examples:

do not broaden into full OCR implementation design
focus on strategic necessity, not parser engineering details
distinguish v1 urgency from long-horizon necessity
separate evidence about current reality from speculative future improvements

Constraints are especially useful for preventing research bloat.

Expected return format

Do not leave this loose.

Good:
Return:

answer to the core question
strongest evidence found
evidence quality assessment
strongest implications for KR
whether the current internal leaning changes
unresolved remainder, if any

Bad:
Give me your findings.

The return format should support direct reintegration into project reasoning.

Success condition

This field answers:
what would make this research worth having done?

Good:
The research should reduce uncertainty enough to say whether PDF support is strategically optional, strategically necessary, or necessary but deferable, and what that implies for KR’s format doctrine.

Bad:
The research should be useful.

If unresolved after research

Some questions will remain only partially resolved.
That possibility should be anticipated.

Good:
If the evidence remains mixed, return the strongest current recommendation, the residual uncertainty, and what additional evidence would be needed for stronger closure.

Bad:
Research more until resolved.

That is too open-ended.

8. Good versus bad request shape
Good request shape
one real question
clearly tied to KR
clear unknown
clear evidence need
clear return format
clear consequence
Bad request shape
vague topic
no decision linkage
no defined unknown
no evidence standard
no scope boundary
no useful return shape

The future production project should refuse to emit bad research requests.

9. Research request readiness checklist

Before issuing a deep-research request, the project should ask:

Have we already done enough internal reasoning to know the actual question?
Is this truly evidence-gated?
Does the answer materially affect KR?
Do we know what kind of evidence we need?
Do we know what kind of answer would help?
Is the scope constrained enough to avoid waste?
Would the result change a real decision, artifact, or aspect state?

If several answers are no, the request should be hardened before being used.

10. Relationship to unresolved questions

When a research request is driven by an unresolved question, the request should ideally point back to that question.

This helps the project preserve the chain:

unresolved question identified
research triggered
evidence gathered
question narrowed, resolved, or reframed
decision or artifact updated

This is one of the main ways the project avoids scattered evidence gathering.

11. Relationship to the deep-research protocol

This template does not decide when research should be used.
That decision belongs to the deep-research and review protocol.

This template answers a different question:

once research is justified, how should the request be shaped?

That separation matters.

12. Relationship to decisions and artifacts

A strong research request should always be linked, directly or indirectly, to:

a decision,
an aspect,
an artifact,
or a blocker.

Research that cannot be connected to one of those is usually not well framed enough for KR.

This rule prevents “research theater.”

13. Anti-failure rules

The future KR production project must avoid the following research-request failures.

13.1 Topic-not-question failure

The request names a topic, not a resolvable question.

13.2 No-stakes failure

The request does not explain why KR needs the answer.

13.3 Unknown-blur failure

The request does not clearly say what is actually unknown.

13.4 Evidence-blur failure

The request does not distinguish strong from weak evidence.

13.5 No-return-shape failure

The research comes back in a form too vague to use.

13.6 Research-theater failure

The request is broad or impressive-looking but not decision-linked.

13.7 Boundary-failure

The request is so broad that it leaks into unrelated domains.

13.8 Internal-reasoning-laziness failure

Research is requested before the project has even clarified the real problem internally.

These failures must be actively resisted.

14. Success condition

This template is successful only if the future KR production project can use it to generate research requests that are:

decision-linked
tightly framed
evidence-aware
efficient
useful on return
and integrated into KR’s broader strategic hardening process

That is the intended standard.

# KR Production Startup Checklist

## 1. Purpose

This checklist defines the exact practical steps for starting the real KR production project in ChatGPT Pro.

Its purpose is to make launch:
- procedural
- low-ambiguity
- low-stress
- consistent
- resistant to setup mistakes

This checklist is not a replacement for the bootstrap protocol.
It is the practical execution companion to it.

The bootstrap protocol explains the logic.
This checklist is the short-form action sequence.

---

## 2. Use rule

Use this checklist only after the pre-work project has reached launch readiness.

Do not use this checklist to begin a half-prepared production project.

If any major prerequisite is missing, stop and return to pre-work rather than improvising.

---

## 3. Pre-launch readiness check

Before doing anything else, confirm all of the following.

### Mandatory readiness items
- [ ] The pre-work project has produced the core operating-system files.
- [ ] The final global custom instructions are ready.
- [ ] The final production-project instructions are ready.
- [ ] The initial upload bundle has been selected.
- [ ] The upload order is decided.
- [ ] The first production aspect is selected.
- [ ] The first production opener prompt is ready.
- [ ] The production launch has passed pre-launch validation sufficiently.
- [ ] No critical setup weakness remains unresolved.

If any box is unchecked, do not launch yet.

---

## 4. Account-level setup

### Step 1 — Buy or activate ChatGPT Pro
- [ ] ChatGPT Pro is active on the account.

### Step 2 — Set the global custom instructions
- [ ] Paste the finalized global custom instructions into account settings.
- [ ] Confirm they are saved correctly.
- [ ] Confirm they are the intended generic baseline, not overloaded with KR-specific doctrine.

### Step 3 — Clear mental separation
- [ ] Confirm that the real KR production project will be a fresh dedicated project.
- [ ] Confirm that setup work is not going to be mixed into the production project casually.

---

## 5. Create the production project

### Step 4 — Create a new project
- [ ] Create a fresh project for KR production work.
- [ ] Use the intended project name.

### Recommended project name
`KR — Strategic Command`

### Step 5 — Choose the intended project memory mode
- [ ] Use the intended isolated/self-contained memory behavior at project creation.
- [ ] Confirm this is a fresh project, not a recycled general-purpose one.

### Step 6 — Paste the production-project instructions
- [ ] Paste the finalized production-project instructions immediately.
- [ ] Confirm they are saved before opening real work chats.

Do not begin production chats before this step is complete.

---

## 6. Upload the initial file bundle

### Step 7 — Upload repo grounding files first
Upload in this order:
- [ ] `.kr/README.md`
- [ ] `.kr/CHARTER.md`
- [ ] `.kr/ACTIVE.md`
- [ ] `.kr/HANDOFF.md`

Goal:
Give the production project current KR reality and control-plane law first.

### Step 8 — Upload production operating-system files second
Upload the core operating files:
- [ ] `ROLE_MODEL_AND_AUTHORITY_SYSTEM.md`
- [ ] `CHAT_WORKFLOW_PROTOCOL.md`
- [ ] `DEEP_RESEARCH_AND_REVIEW_PROTOCOL.md`
- [ ] `KR_MASTER_ASPECT_MAP.md`
- [ ] `REPO_ARTIFACT_SYSTEM.md`
- [ ] `EXECUTION_HANDOFF_SYSTEM.md`
- [ ] `PRODUCTION_BOOTSTRAP_PROTOCOL.md`
- [ ] `DECISION_LOG_SCHEMA.md`
- [ ] `ASPECT_STATE_MODEL.md`
- [ ] `FILE_NAMING_AND_PLACEMENT_RULES.md`
- [ ] `ACCEPTANCE_CRITERIA_FRAMEWORK.md`
- [ ] `CODEX_TASK_PACKET_TEMPLATE.md`
- [ ] `ARTIFACT_REGISTRY_SCHEMA.md`
- [ ] `UNRESOLVED_QUESTIONS_REGISTRY_SCHEMA.md`
- [ ] `ASPECT_REGISTRY_SCHEMA.md`
- [ ] `DEEP_RESEARCH_REQUEST_TEMPLATE.md`
- [ ] `PRODUCTION_STARTUP_CHECKLIST.md`

### Step 9 — Upload core substantive KR files third
Upload the most relevant substantive files for early work:
- [ ] key architecture overview file
- [ ] excerpting spec
- [ ] taxonomy spec
- [ ] important evaluation/rubric file(s)
- [ ] any file directly needed for the first chosen aspect

Do not dump large amounts of low-authority material into the project at launch.

---

## 7. Post-upload sanity check

### Step 10 — Confirm the project has the right context shape
Check mentally:

- [ ] The project has current KR repo reality.
- [ ] The project has its operating-system files.
- [ ] The project has enough substantive context to begin.
- [ ] The project does not have excessive clutter.
- [ ] The project’s center of gravity is clear.

If the upload set feels noisy or muddled, fix that before the first chat.

---

## 8. Start the first production chat

### Step 11 — Open the first real production chat
- [ ] Start a fresh chat inside the production project.
- [ ] Use the finalized first aspect opener prompt.
- [ ] Do not begin with a generic broad prompt.

### Recommended first aspect
`A2 — V1 proof doctrine`

### Step 12 — Paste the first opener exactly or near-exactly
- [ ] Use the prepared first-chat opener.
- [ ] Keep the scope narrow.
- [ ] Do not combine multiple aspects in the first chat.

---

## 9. First-chat validation

### Step 13 — Evaluate whether the project behaves correctly
After the first serious response, check:

#### Behavioral checks
- [ ] It took initiative.
- [ ] It did not wait passively for guidance.
- [ ] It minimized unnecessary questions.
- [ ] It behaved like a thinking owner, not a generic assistant.

#### Scope checks
- [ ] It stayed within one aspect.
- [ ] It did not broaden into a vague full-project review.
- [ ] It worked on a bounded subtask rather than everything at once.

#### Quality checks
- [ ] It surfaced real assumptions.
- [ ] It showed critique or hardening behavior.
- [ ] It distinguished current state, v1, and longer-horizon ambition where relevant.
- [ ] It aimed toward durable outputs.

#### Workflow checks
- [ ] The response made it plausible that you could say “continue.”
- [ ] The thread feels like it can progress subtask by subtask.
- [ ] The project rhythm feels usable.

If several of these fail, do not ignore it.

---

## 10. If the first chat behaves well

### Step 14 — Continue the thread properly
- [ ] Use short continuation prompts as intended.
- [ ] Prefer prompts like:
  - continue
  - go deeper
  - harden this
  - critique this
  - convert this into a file
  - test edge cases

### Step 15 — Stay disciplined
- [ ] Keep the chat on the same aspect.
- [ ] Open new chats only when the aspect truly changes or a sub-question deserves separation.
- [ ] Preserve the production project as the main KR worksite.

At this point, the production project is effectively live.

---

## 11. If the first chat behaves badly

### Step 16 — Do not silently tolerate weak launch behavior
If the project behaves like:
- a passive assistant
- an over-broad summarizer
- a repo servant
- a question-heavy collaborator
- a non-artifact-producing conversationalist

then do not just continue and hope it improves.

### Step 17 — Diagnose the failure
- [ ] Identify the exact failure pattern.
- [ ] Identify whether the likely cause is:
  - weak project instructions
  - weak upload bundle
  - weak opener prompt
  - missing operating file
  - launch clutter
  - setup drift

### Step 18 — Correct deliberately
- [ ] Update the relevant setup element.
- [ ] Re-run the launch logic cleanly.
- [ ] Do not let the production project become a second pre-work project indefinitely.

---

## 12. Early production discipline

Once the project is live, maintain these early-stage rules.

### Rule A
- [ ] One aspect per chat.

### Rule B
- [ ] One subtask at a time within the chat.

### Rule C
- [ ] Prefer durable outputs over vague discussion.

### Rule D
- [ ] Use deep research when truly evidence-gated, not decoratively.

### Rule E
- [ ] Do not hand execution work to Codex until the task is genuinely hardened enough.

### Rule F
- [ ] Keep the project self-contained and clean.

These rules are especially important in the first few production chats because early rhythm tends to persist.

---

## 13. Minimum success condition for launch

The production startup should count as successful only if all of the following are true:

- [ ] The production project exists with the intended instructions.
- [ ] The intended file bundle is in place.
- [ ] The first aspect chat has been started.
- [ ] The first chat behaved substantially as intended.
- [ ] The project can now continue through disciplined aspect-by-aspect work.
- [ ] Further foundational setup is no longer the main bottleneck.

If these are not true, launch is not yet complete.

---

## 14. Final practical reminder

This checklist is for execution discipline, not reflection.

When launch time comes:
- follow it cleanly
- do not improvise unnecessarily
- do not overload the project at birth
- and do not confuse “created the project” with “successfully launched the system”

The real goal is not project creation.

The real goal is a production KR project that is immediately capable of high-rigor strategic work.

# KR Pre-Launch Validation Protocol

## 1. Purpose

This document defines how the pre-work project should validate the future KR production project before launch.

Its purpose is to ensure that the production project is not created on the basis of:
- attractive setup theory
- incomplete operating files
- untested assumptions about behavior
- weak launch sequencing
- or false confidence in the instruction stack

This protocol exists to answer one question with discipline:

“Is the KR production operating system actually ready to go live?”

The answer must not be guessed.
It must be validated.

---

## 2. Core principle

The production project should not be the place where the operating system is first stress-tested.

The pre-work project must test the operating system before launch by simulating:
- startup behavior
- aspect-chat behavior
- continuation behavior
- artifact-conversion behavior
- review behavior
- escalation behavior
- handoff behavior

The goal is not perfect certainty.
The goal is strong enough validation that launch begins from a disciplined, low-regret state.

---

## 3. Validation philosophy

Validation should test not only whether the setup exists, but whether it works operationally.

That means the protocol must test all of the following:

- behavioral correctness
- workflow correctness
- scope discipline
- durability of outputs
- question minimization
- research invocation discipline
- review discipline
- artifact-generation discipline
- launch cleanliness
- maintainability of the operating system

The future production project must be validated as a working system, not merely a document bundle.

---

## 4. Validation categories

The pre-launch validation should cover six categories.

### V1 — completeness validation
Does the operating system package actually contain everything required for launch?

### V2 — coherence validation
Do the documents fit together without contradiction, overlap confusion, or missing bridges?

### V3 — behavior validation
Would the production project actually behave like the intended thinking owner?

### V4 — workflow validation
Would the one-aspect-per-chat / one-subtask-at-a-time model actually work in practice?

### V5 — artifact and execution validation
Would the project reliably convert reasoning into durable files and bounded execution packets?

### V6 — launch validation
Would the user actually be able to create the production project cleanly and start real work immediately?

All six categories must pass sufficiently before launch.

---

## 5. Validation sequence

The pre-launch validation should happen in the following order:

### Step 1 — document completeness check
Confirm that all essential files exist.

### Step 2 — internal consistency check
Check whether the files fit together coherently.

### Step 3 — simulated production launch check
Run the production startup sequence conceptually from beginning to first chat.

### Step 4 — first-chat simulation
Simulate the first real production chat and inspect its behavior.

### Step 5 — continuation simulation
Test whether the “continue” workflow behaves properly.

### Step 6 — escalation and research simulation
Test whether the system escalates correctly and invokes research only when justified.

### Step 7 — artifact-conversion simulation
Test whether the system turns hardened conclusions into the right kinds of repo artifacts.

### Step 8 — execution-handoff simulation
Test whether the system can generate a good task packet for bounded implementation.

### Step 9 — launch verdict
Decide whether the production system is:
- ready to launch
- ready with minor corrections
- or not ready

This sequence should be followed deliberately.

## 7. Completeness validation

## Purpose
Check whether the production operating system is materially complete.

## Questions
- Do all required constitutional files exist?
- Do all required operating files exist?
- Do the finalized instructions exist?
- Does the bootstrap protocol exist?
- Does the first chat opener exist?
- Does the initial upload bundle exist?
- Are any critical support files still only implicit?

## Pass condition
Completeness validation passes only if no essential production-launch component is missing.

## Failure signals
- a core protocol still exists only in chat memory
- the instruction stack is not finalized
- the launch sequence cannot be run from written materials alone
- a key behavior depends on unwritten assumptions

---

## 8. Coherence validation

## Purpose
Check whether the files form one coherent operating system rather than a pile of individually good documents.

## Questions
- Do role rules align with instruction rules?
- Do instruction rules align with workflow rules?
- Do workflow rules align with review rules?
- Do review rules align with handoff rules?
- Does the aspect map align with the intended chat model?
- Does the repo-artifact system align with the control-plane boundary?
- Does the bootstrap protocol align with the instruction stack and upload bundle?

## Pass condition
Coherence validation passes only if the files support one another cleanly and no major contradiction or missing bridge exists.

## Failure signals
- two files assign different authority boundaries
- one file encourages breadth while another requires strict aspect narrowness
- one file implies research-heavy behavior while another discourages it indiscriminately
- a file assumes a workflow that no other file actually defines

---

## 9. Behavior validation

## Purpose
Check whether the production system would actually make ChatGPT behave like KR’s intended thinking owner.

## Questions
- Does the system push the model toward initiative?
- Does it minimize unnecessary questions?
- Does it authorize challenge toward the repo without encouraging chaos?
- Does it create the right balance between ownership and humility?
- Does it push the model to create durable artifacts rather than only discuss?
- Does it prevent both passive-assistant and reckless-owner failure modes?

## Pass condition
Behavior validation passes only if the system would predictably cause the model to:
- lead strongly
- ask little
- challenge appropriately
- stay honest about uncertainty
- produce structure

## Failure signals
- the system is likely to produce timid, overly deferential behavior
- the system is likely to produce ungrounded overreach
- the system does not clearly authorize artifact creation
- question minimization is stated aspirationally but not operationally

---

## 10. Workflow validation

## Purpose
Check whether the one-aspect-per-chat and one-subtask-at-a-time method is operationally usable.

## Questions
- Can a chat open cleanly around one aspect?
- Can the model advance through bounded subtasks?
- Does “continue” have a stable meaning?
- Does the workflow resist silent scope drift?
- Does the workflow know when to close, split, or escalate?
- Does the system avoid both broadening and micro-fragmentation?

## Pass condition
Workflow validation passes only if the project could plausibly run for many chats without collapsing into chaos or friction.

## Failure signals
- chat boundaries are too vague
- “continue” would likely trigger recap instead of progress
- there is no clear closure logic
- the system would likely open too many chats or too few
- durable output is not naturally integrated into the workflow

---

## 11. Research and review validation

## Purpose
Check whether the system uses deep research and review at the right times and in the right way.

## Questions
- Does it distinguish internal design questions from evidence-gated questions?
- Are deep-research triggers explicit enough?
- Are there enough critique passes for major decisions?
- Is uncertainty handled explicitly?
- Are confidence and evidence kept separate?
- Does the system avoid both under-research and research-as-decoration?

## Pass condition
Research and review validation passes only if the system would use Pro’s strongest capabilities in a disciplined, high-value way.

## Failure signals
- the system would likely invoke research too often without real need
- the system would likely avoid research where it is clearly necessary
- the review protocol sounds rigorous but lacks real adversarial structure
- hardening completion is too vague to operationalize

---

## 12. Artifact-system validation

## Purpose
Check whether the system would reliably produce the right durable repo artifacts.

## Questions
- Does it know when a conclusion deserves a file?
- Does it know when to update an existing file instead?
- Does it distinguish doctrine from protocol, registry, schema, template, and execution artifact?
- Does it keep the control plane clean?
- Does it know when to use prose versus structured data?
- Does it prevent file sprawl?

## Pass condition
Artifact-system validation passes only if the future production project would reliably transform important thinking into clean repo structure.

## Failure signals
- too many new-file triggers
- unclear artifact-class boundaries
- no structured artifact proposal format
- registry and doctrine logic are muddled
- control-plane pollution risk remains high

---

## 13. Execution-handoff validation

## Purpose
Check whether the production system can actually generate safe, bounded work for Codex or another execution agent.

## Questions
- Does it know when a task is execution-ready?
- Is the task packet sufficiently explicit?
- Are acceptance criteria real, not decorative?
- Are discretion boundaries visible?
- Is returned-work review defined properly?
- Can the system distinguish bad execution from bad packet design?

## Pass condition
Execution-handoff validation passes only if the system can translate hardened strategy into bounded implementation work without strategic slippage.

## Failure signals
- packets would still require the worker to invent doctrine
- acceptance criteria are weak
- review after return is underdefined
- scope boundaries are too soft
- execution and strategy remain insufficiently separated

---

## 14. Launch simulation validation

## Purpose
Check whether the user could actually instantiate the production project cleanly using only the written bootstrap package.

## Questions
- Is the project creation sequence explicit enough?
- Is the upload bundle clear enough?
- Is the upload order clear enough?
- Are the project instructions ready to paste?
- Is the first aspect clearly chosen?
- Is the first chat opener good enough to trigger the intended behavior?
- Does the validation checklist make launch success inspectable?

## Pass condition
Launch simulation validation passes only if a user could follow the bootstrap package step by step without needing a second setup phase.

## Failure signals
- the bootstrap depends on unwritten judgment
- the user still needs to improvise key setup choices
- the first chat is under-specified
- success criteria for launch are vague

---

## 15. Simulation protocol

The pre-work project should validate the future production system by running structured simulations.

Three simulations are mandatory:

### S1 — first-chat simulation
Simulate the first production chat using the actual first-chat opener.

The simulation should test:
- whether the project adopts the intended role
- whether it stays within one aspect
- whether it advances through bounded subtasks
- whether it produces durable output
- whether it minimizes unnecessary questions

### S2 — “continue” simulation
Simulate at least one continuation turn after the first chat response.

The simulation should test:
- whether continuity is preserved
- whether the next bounded subtask is chosen intelligently
- whether the system deepens rather than restarts
- whether the chat remains within scope

### S3 — escalation simulation
Simulate a moment where:
- real user preference is needed,
- or evidence is needed,
- or an irreversible choice appears.

The simulation should test:
- whether the system escalates correctly
- whether it asks a precise question
- whether it still gives a strong recommendation
- whether it avoids dumping the thinking burden back on the user

These simulations should be treated as real tests, not symbolic gestures.

---

## 16. Edge-case validation

The system should also be checked against difficult edge cases.

Recommended edge cases:

### E1 — repo narrowness challenge
Would the system notice a strategically valuable missing capability that the current repo does not mention?

### E2 — premature research temptation
Would the system avoid invoking research for a purely internal protocol-design issue?

### E3 — domain uncertainty
Would the system avoid bluffing on an Islamic-science-sensitive question?

### E4 — artifact overproduction
Would the system avoid creating five files when one update or registry entry would do?

### E5 — execution under-hardening
Would the system refuse to hand off a fuzzy task packet?

### E6 — chat drift
Would the system stop itself from silently sliding into a new aspect?

### E7 — user-minimization stress
Would the system continue moving forward under a sequence of simple prompts like “continue”?

These edge cases should be used to expose hidden weakness before launch.

---

## 17. Validation scoring model

The pre-work project should rate each validation category using one of the following:

### G0 — fail
Serious weakness. Launch should not proceed.

### G1 — weak
Usable in fragments, but not trustworthy enough for launch.

### G2 — acceptable
Good enough to launch with awareness of remaining limitations.

### G3 — strong
Robust and well-designed.

### G4 — excellent
Exceptionally well-hardened and low-regret.

The categories to score:
- completeness
- coherence
- behavior
- workflow
- research/review
- artifact system
- execution handoff
- launch simulation

The production project should not launch if any core category is G0.
It should generally not launch if multiple core categories are only G1.

---

## 18. Launch verdict classes

At the end of validation, the pre-work project should issue one of three verdicts.

### L1 — launch approved
The operating system is sufficiently strong to begin production work.

### L2 — launch approved with corrections
The system is good enough to launch, but a small number of explicit corrections should be made immediately or very early.

### L3 — launch held
The operating system still has serious weaknesses and should not be used as the production foundation yet.

This verdict should be explicit.
The pre-work project should not end in vague optimism.

---

## 19. Correction protocol after failed validation

If validation finds a serious problem, the pre-work project should not respond with vague “we should improve this.”

It should identify:
- the exact weakness
- which file or protocol caused it
- why the weakness matters
- what correction is needed
- whether that correction is local or systemic
- whether revalidation is required afterward

Corrections should be explicit and targeted.

---

## 20. Validation-output format

At the end of a full validation cycle, the pre-work project should produce a structured validation memo.

### Validation memo format

- validation date:
- version of setup package reviewed:
- categories tested:
- category scores:
- strongest strengths:
- strongest weaknesses:
- edge cases failed:
- corrections required:
- corrections recommended:
- launch verdict:
- if approved, launch conditions:
- if held, exact blockers to resolve:

This memo becomes the final decision instrument for whether production launch should occur.

---

## 21. Anti-failure rules

The pre-work project must avoid the following validation failures.

### 21.1 Checkbox validation failure
The files exist, so the system is treated as ready without testing behavior.

### 21.2 Courtesy validation failure
The setup is treated as good because it sounds impressive, not because it was stress-tested.

### 21.3 Theory-only validation failure
The system is validated conceptually but never simulated operationally.

### 21.4 Under-edge-case validation failure
Only easy scenarios are tested.

### 21.5 Vague-verdict failure
The validation ends without a clear launch decision.

### 21.6 No-correction-path failure
Weaknesses are identified, but no explicit correction path is given.

### 21.7 Launch-fatigue failure
The system is launched early simply because the pre-work phase feels long.

These failures must be actively resisted.

---

## 22. Completion rule for the pre-work project

The pre-work project should not be considered complete merely because the documents exist.

It is complete only when:
- the operating system package exists
- the bootstrap package exists
- validation has been run
- a launch verdict has been issued
- any required corrections have been applied
- the production project is genuinely ready to begin as a clean, self-contained work environment

This is the real completion bar.

---

## 23. Success condition

The pre-launch validation protocol is successful only if it prevents the KR production project from beginning with avoidable setup flaws.

It should make launch:
- earned
- explicit
- evidence-based
- operationally justified
- low-regret

That is the intended role of validation.
